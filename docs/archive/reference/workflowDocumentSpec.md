# 워크플로우 문서 규격

> Status: active
> Created: 2026-04-01
> 목적: 오케스트레이션 파이프라인에서 생성되는 모든 문서의 디렉토리 구조, 파일명 규칙, 내용 규격을 정의

---

## 1. 디렉토리 구조

```
{project_root}/
└── docs/
    ├── plans/                    ← Plan 문서 (계획서 + 작업지시서)
    │   ├── {slug}.md             ← Plan 본문
    │   ├── {slug}-review-r1.md   ← Review 보고서 (라운드 1)
    │   ├── {slug}-review-r2.md   ← Review 보고서 (라운드 2, rework 후)
    │   └── {slug}-result.md      ← 구현 결과 보고서
    └── agents/                   ← 에이전트 역할 템플릿 (프로젝트 scaffold 시 생성)
        ├── architect.md
        ├── developer.md
        └── reviewer.md
```

### 파일 소유 규칙

| 디렉토리 | 생성 주체 | 수정 주체 | 비고 |
|----------|----------|----------|------|
| `docs/plans/*.md` | tunaFlow (자동) | tunaFlow (자동) | 사용자/에이전트가 직접 수정하지 않음 |
| `docs/agents/*.md` | tunaFlow (scaffold) | 사용자 (커스터마이징 가능) | ContextPack에서 role 기반 로딩 |

## 2. 파일명 규칙

### 2.1 Slug 생성

```
입력: "tunaInsight — 멀티 에이전트 GitHub 저장소 비교 분석 서비스"
출력: "tunainsight-멀티-에이전트-github-저장소-비교-분석-서비스"
```

규칙:
- 영문 → 소문자
- 한글 → 유지
- 공백/특수문자 → `-`
- 연속 `-` 제거
- 최대 80자

### 2.2 파일명 패턴

| 파일 | 패턴 | 예시 |
|------|------|------|
| Plan 문서 | `{slug}.md` | `api-리팩토링.md` |
| Review 보고서 | `{slug}-review-r{N}.md` | `api-리팩토링-review-r1.md` |
| 구현 결과 보고서 | `{slug}-result.md` | `api-리팩토링-result.md` |

- `{N}` = review 라운드 번호 (1부터 시작, rework 후 재리뷰 시 증가)

## 3. 문서 규격

### 3.1 Plan 문서 (`{slug}.md`)

```markdown
# {Plan 제목}

> Phase: {phase} | Status: {status} | Revision: {revision}
> Generated by tunaFlow

---

## Description

{Plan 설명}

## Expected Outcome

{기대 결과}

## Subtasks

### 1. {Subtask 제목}

- **Status**: {todo|approved|in_progress|done|abandoned}
- **Owner**: {에이전트 이름}

#### 작업 지시

{상세 설계 내용 — how}

### 2. {Subtask 제목}

...

## Revision History

- `{YYYY-MM-DD HH:MM}` {event_type} ({actor}) — {detail}
- `{YYYY-MM-DD HH:MM}` {event_type} ({actor}) — {detail}
```

**생성 시점**: Plan 승격 시
**업데이트 시점**: 상세 설계 병합, subtask revision 병합, Plan 수정 병합
**업데이트 하지 않는 시점**: phase 전환만 있을 때 (내용 변경 없음)

### 3.2 Review 보고서 (`{slug}-review-r{N}.md`)

```markdown
# Review Report: {Plan 제목} — Round {N}

> Verdict: {pass|fail|conditional}
> Reviewer: {reviewer 에이전트 목록}
> Date: {YYYY-MM-DD HH:MM}
> Plan Revision: {revision at review time}

---

## Verdict

**{pass|fail|conditional}**

## Findings

1. {finding 1}
2. {finding 2}
...

## Recommendations

1. {recommendation 1}
2. {recommendation 2}
...

## Subtask Verification

| # | Subtask | 구현 여부 | 비고 |
|---|---------|----------|------|
| 1 | {title} | ✅/❌ | {comment} |
| 2 | {title} | ✅/❌ | {comment} |

## Test Results

- **Runner**: {cargo|vitest|jest}
- **Passed**: {N}
- **Failed**: {N}
- **Skipped**: {N}

```raw
{테스트 출력 (최대 2000자)}
```
```

**생성 시점**: Review 완료 시 (verdict 마커 감지 후 사용자가 verdict 처리 시)
**업데이트**: 없음 (불변 — 라운드별 새 파일)

### 3.3 구현 결과 보고서 (`{slug}-result.md`)

```markdown
# Implementation Result: {Plan 제목}

> Developer: {developer 에이전트}
> Branch: {implementation branch label}
> Date: {YYYY-MM-DD HH:MM}
> Plan Revision: {revision}

---

## Summary

{구현 요약 — Developer의 impl-complete 시점 대화 압축}

## Subtask Results

### 1. {Subtask 제목}

- **Status**: {done|partial|skipped}
- **변경 파일**: {파일 목록}
- **비고**: {특이사항}

### 2. {Subtask 제목}

...

## Known Issues

- {알려진 이슈 1}
- {알려진 이슈 2}
```

**생성 시점**: Developer impl-complete 감지 후, Review 시작 전
**업데이트**: rework 후 재구현 시 덮어쓰기

## 4. 생성 규칙 (시점 정리)

| 워크플로우 이벤트 | 동작 | 대상 문서 |
|------------------|------|----------|
| Plan 승격 (Chat → Plan) | 생성 | `{slug}.md` |
| 상세 설계 병합 | 업데이트 | `{slug}.md` |
| Subtask revision 병합 | 업데이트 | `{slug}.md` |
| Plan 수정 병합 (Dev → Architect → 병합) | 업데이트 | `{slug}.md` |
| Dev impl-complete 감지 | 생성 | `{slug}-result.md` |
| Review verdict 처리 | 생성 | `{slug}-review-r{N}.md` |
| Rework → 재구현 → impl-complete | 덮어쓰기 | `{slug}-result.md` |
| Rework → 재리뷰 → verdict | 생성 (N+1) | `{slug}-review-r{N+1}.md` |
| Phase 전환만 (내용 변경 없음) | 없음 | — |

## 5. ContextPack 주입 규칙

| 문서 | 주입 대상 | 주입 조건 |
|------|----------|----------|
| `{slug}.md` | 모든 에이전트 | active plan이 있을 때 |
| `{slug}-result.md` | Reviewer | review phase일 때 |
| `{slug}-review-r{N}.md` | Developer (rework 시) | rework phase일 때, 최신 review만 |
| `docs/agents/{role}.md` | 해당 역할 에이전트 | 항상 |

---

## 6. 에이전트 역할별 문서 참조

| 역할 | 읽는 문서 | 생성하는 문서 |
|------|----------|-------------|
| **Architect** | Plan 문서 | Plan 문서 (plan-proposal 마커) |
| **Developer** | Plan 문서, Review 보고서 (rework 시) | 구현 결과 (impl-complete 마커) |
| **Reviewer** | Plan 문서, 구현 결과 보고서 | Review 보고서 (review-verdict 마커) |

---

## 7. 절대 하지 말 것

1. **사용자/에이전트가 Plan 문서를 직접 편집** — tunaFlow가 DB 기반으로 자동 생성/업데이트
2. **Review 보고서를 수정** — 불변, 라운드별 새 파일
3. **문서 없이 구현 시작** — Plan 문서가 SSOT, 문서 없으면 Developer가 작업 기준 없음
4. **마커 없이 문서 생성 시도** — 모든 문서 생성은 마커 감지 → tunaFlow 자동 처리
5. **docs/plans/ 밖에 워크플로우 문서 생성** — 모든 파이프라인 산출물은 docs/plans/ 아래
