# Computer Use E2E 테스트

> Status: idea
> Created: 2026-04-15
> 관련: Claude Code Computer Use (v2.1.85+)

---

## 배경

tunaFlow는 Tauri 데스크톱 앱이라 Playwright/Cypress 같은 웹 테스트 프레임워크로 e2e 커버가 안 된다. 기존 smoke test는 깨진 상태이고, Tauri WebView를 직접 제어하는 테스트는 세팅 비용이 크다.

Claude Code의 Computer Use(내장 MCP 서버)는 화면에 보이는 그대로 조작하므로 Tauri든 네이티브든 상관없이 동작한다. 대화형 Claude Code 세션에서 시나리오를 프롬프트로 실행하면 실제 UI를 클릭·입력·스크린샷하며 검증할 수 있다.

---

## 실행 방법

```bash
# 1. tunaFlow 실행
npm run tauri dev

# 2. Claude Code 대화형 세션에서
/mcp   # computer-use 활성화 (최초 1회)

# 3. 아래 시나리오를 프롬프트로 입력
```

**사전 조건:**
- macOS
- Claude Code v2.1.85+
- Pro/Max 구독
- 접근성 + 화면 녹화 권한 (최초 1회 승인)

## 테스트 시나리오

### S0. WS 모드 기본 동작 검증 (신규 — 즉시 실행 가능)

> `--sdk-url` WS 세션으로 교체 후 메인챗이 정상 동작하는지 확인하는 회귀 시나리오.
> 기존 `-p` 모드와 달라진 5가지 포인트를 순서대로 체크한다.

```
tunaFlow를 실행한다 (npm run tauri dev).

[체크 1 — 모드 인디케이터]
메인챗 입력창의 모드 바를 확인한다.
claude 엔진이 선택된 상태에서 "WS" 파란 뱃지가 표시되는지 확인한다.
뱃지에 마우스를 올려 "--sdk-url WS 세션 (슬래시 커맨드 지원)" 툴팁이 뜨는지 확인한다.
스크린샷을 찍는다.

[체크 2 — 첫 번째 메시지 스트리밍]
채팅 입력창에 "숫자 1부터 5까지를 한 줄씩 출력하는 파이썬 코드를 짧게 써줘" 를 입력하고 전송한다.
응답이 스트리밍으로 들어오는지 확인한다.
응답에 ANSI 이스케이프 코드 잔재(예: ^[[0m, ESC[ 같은 문자)가 없는지 확인한다.
응답이 완료될 때까지 기다린다.
스크린샷을 찍는다.

[체크 3 — 세션 지속 (두 번째 메시지)]
같은 대화에서 추가로 "방금 코드에 주석을 추가해줘" 를 입력하고 전송한다.
이전 대화 맥락을 기억하고 올바르게 응답하는지 확인한다.
(세션이 유지되면 subprocess 재생성 없이 응답해야 한다)
스크린샷을 찍는다.

[체크 4 — 슬래시 커맨드]
입력창에 "/compact" 를 입력하고 전송한다.
슬래시 커맨드가 인식되어 처리되는지 확인한다.
에러 없이 동작하면 성공, 에러가 뜨면 스크린샷을 찍고 에러 메시지를 기록한다.

[체크 5 — 취소 동작]
입력창에 "피보나치 수열 1000번째 항까지 전부 나열해줘" 를 입력하고 전송한다.
스트리밍이 시작되면 즉시 Cancel 버튼을 클릭한다.
스트리밍이 중단되고 UI가 정상 상태로 돌아오는지 확인한다.
스크린샷을 찍는다.

모든 체크가 완료되면 각 체크의 pass/fail을 정리해서 보고한다.
```

**예상 실패 지점 및 대응:**

| 실패 케이스 | 원인 추정 | 확인 방법 |
|-------------|-----------|-----------|
| WS 뱃지 미표시 | `get_claude_mode` 커맨드 미응답 | 개발자 도구 콘솔 에러 확인 |
| 응답 없음 (30초 timeout) | claude WS 연결 실패 | `npm run tauri dev` 터미널 로그 확인 |
| ANSI 문자 잔재 | 스트림 파싱 오류 | 응답 원문 텍스트 복사해서 확인 |
| 슬래시 커맨드 무반응 | WS 세션 아직 미지원 경로 | 예상 범위 내 — 기록만 |
| Cancel 후 UI 먹통 | interrupt 메시지 미전달 | 앱 재시작 후 재현 여부 확인 |

### Tier 1: 릴리즈 전 필수 (4개)

**S1. 풀사이클 워크플로우**
```
tunaFlow를 열고 테스트 프로젝트를 선택한다.
채팅에서 "간단한 유틸 함수를 만들어줘"를 입력한다.
Plan 제안이 오면 Approve 버튼을 클릭한다.
Dev Branch가 드로어로 열리는지 확인한다.
Developer 에이전트가 실행 완료될 때까지 기다린다.
Review Branch가 자동 생성되는지 확인한다.
각 단계를 스크린샷한다.
```

**S2. Branch 생성 → adopt**
```
메인 채팅에서 메시지를 하나 보낸다.
해당 메시지를 우클릭하고 "Branch 분기"를 클릭한다.
드로어에서 Branch가 열리는지 확인한다.
Branch에서 메시지를 하나 보내고 응답을 기다린다.
Adopt 버튼을 클릭한다.
메인 대화에 요약 메시지가 삽입되었는지 확인한다.
```

**S3. 드로어/사이드바 레이아웃**
```
사이드바에서 Branches 섹션을 확인한다.
Branch를 클릭해서 드로어가 열리는지 확인한다.
드로어의 📌 버튼을 클릭해서 고정 패널 모드로 전환한다.
CenterPanel과 50:50으로 분할되는지 확인한다.
다른 대화로 이동하면 고정이 해제되는지 확인한다.
각 상태를 스크린샷한다.
```

**S4. 라이트/다크 모드 전환**
```
Settings를 열고 테마를 라이트로 변경한다.
채팅 화면, 사이드바, 드로어가 라이트 모드로 렌더링되는지 확인한다.
다크로 다시 변경하고 동일하게 확인한다.
각 상태를 스크린샷한다.
```

### Tier 2: 주요 기능 검증 (3개)

**S5. Settings 변경 → 에이전트 반영**
```
Settings > Profile에서 이름과 bio를 입력한다.
Settings > Runtime에서 Context Mode를 Full로 변경한다.
채팅에서 메시지를 보내고 에이전트가 응답하는지 확인한다.
```

**S6. Roundtable 생성 및 실행**
```
사이드바에서 "New Roundtable" 버튼을 클릭한다.
Sequential 모드를 선택하고 주제를 입력한다.
RT가 실행되고 참가자들이 순차적으로 응답하는지 확인한다.
```

**S7. Insight 분석 실행**
```
Insight 탭을 클릭한다.
"Run Analysis" 버튼을 클릭한다.
진행 로그가 표시되고 분석이 완료되는지 확인한다.
결과 카드가 렌더링되는지 확인한다.
```

## 비결정성 대응

에이전트 응답이 매번 다르므로 exact assertion이 불가능하다. 검증 기준은:

- **구조적 검증**: "드로어가 열렸는가", "버튼이 존재하는가", "스크린샷에 에러 상태가 없는가"
- **상태 전이 검증**: "Branch가 active → adopted로 변경되었는가"
- **부재 검증**: "에러 토스트가 뜨지 않았는가", "빈 화면이 아닌가"

Computer Use가 스크린샷을 찍고 시각적으로 판단하므로, pixel-exact가 아니라 "이 UI 요소가 보이는가" 수준의 검증이 자연스럽다.

## 테스트 피라미드에서의 위치

```
테스트 계층:
  ├── Unit (cargo test + vitest)     — 매 커밋, CI 필수, 초 단위
  ├── Integration (Tauri command)    — 매 커밋, CI 필수, 초 단위
  └── E2E Smoke (Computer Use)       — 릴리즈 전, 개발자 수동 실행, 분 단위
        ├── Tier 1: 풀사이클/Branch/레이아웃/테마 (필수 4개)
        └── Tier 2: Settings/RT/Insight (주요 3개)
```

Unit/Integration은 빠르고 결정적이라 매 커밋에 돌리고, Computer Use E2E는 느리고 비결정적이지만 **실제 UI를 실제로 클릭**하므로 릴리즈 전 스모크로 수동 실행한다.

## 제약사항

- **macOS 전용**: Computer Use가 macOS에서만 동작
- **Pro/Max 구독 필요**
- **머신 잠금**: Computer Use 실행 중 다른 GUI 작업 불가
- **비결정적**: 에이전트 응답 변동 → Tier 1은 구조적 검증으로 최소화
