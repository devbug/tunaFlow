# tunaFlow 문서 내비게이션 모델

- 작성자: OpenAI Codex
- 작성 시각: 2026-03-30
- 목적: 에이전트와 사람이 전체 `docs/`를 다 읽지 않고도 필요한 문서를 빠르게 찾게 하는 기준

## 핵심 판단

문서가 많아지는 것이 문제라기보다,
문서의 역할과 읽는 순서가 불명확한 것이 문제다.

따라서 해결은 문서 삭제보다 다음 네 가지다.

1. 문서 타입 구분
2. 읽기 순서 고정
3. 인덱스 품질 향상
4. 낡은 문서/초안/아카이브 구분

## 문서 타입

### 1. `reference/`

역할:

- 현재 기준 사실
- SSOT 또는 그에 준하는 문서

예:

- 구현 현황
- 데이터 모델
- 작업 규칙
- 프로젝트 기준 참고 문서

읽는 법:

- 가장 먼저 읽는 문서군

### 2. `plans/`

역할:

- 앞으로 무엇을 할지
- 어떤 순서로 할지
- 비목표와 완료 기준이 무엇인지

읽는 법:

- 현재 작업과 관련 있는 문서만 읽는다
- 전체를 다 읽지 않는다

### 3. `prompts/`

역할:

- Claude 같은 실행 에이전트에게 넘기는 작업 지시문

읽는 법:

- 반드시 대응하는 `plans/` 문서와 함께 읽는다

### 4. `how-to/`

역할:

- 운영 가이드
- 실제 사용 절차

읽는 법:

- 특정 기능을 쓰거나 설정할 때만 읽는다

### 5. `archive/`

역할:

- 더 이상 현재 기준이 아닌 과거 계획
- 기록 보존용

읽는 법:

- 현재 구현 기준 판단에는 우선 사용하지 않는다

## 권장 읽기 순서

### 새 세션 공통

1. `CLAUDE.md`
2. `docs/reference/implementationStatus.md`
3. `docs/reference/codexProjectReference_2026-03-29.md`
4. 현재 작업 관련 `docs/plans/*.md`
5. 해당 실행용 `docs/prompts/*.md`

### 특정 기능 작업

1. 상위 reference 1~2개
2. 관련 plan 1개
3. 관련 prompt 1개
4. 필요 시 how-to 1개

원칙:

- 한 작업에 3~5개 문서를 넘기면 충분하다
- 전체 docs를 전부 읽게 하면 안 된다

## 인덱스 규칙

### index 문서의 역할

각 폴더의 `index.md`는 단순 목록이 아니라:

- 무엇이 기준 문서인지
- 무엇이 현재 유효한지
- 어떤 순서로 읽어야 하는지

를 알려줘야 한다.

### index에 반드시 있어야 할 것

- 짧은 폴더 설명
- 추천 읽기 순서
- 현재 유효/부분 완료/보류 상태
- 아카이브 문서 경고

## 파일명 규칙

### 권장

- 기능명 + 목적 + 날짜
- 예: `agentProfileChatInputBindingPlan_2026-03-29.md`

### 피해야 할 것

- 의미 없는 일반명
- 버전/날짜 없이 비슷한 이름 반복
- 현재 문서와 과거 문서가 같은 이름으로 공존

## 문서 상태 라벨

문서 첫머리에 가능하면 아래 중 하나를 둔다.

- `상태: 초안`
- `상태: 진행 중`
- `상태: 부분 완료`
- `상태: 완료`
- `상태: 아카이브`

이 라벨이 없으면 에이전트가 현재 기준 문서인지 판단하기 어렵다.

## 에이전트 친화 규칙

에이전트가 찾기 쉬우려면:

1. 한 문서에 한 목적만 담는다
2. 목적 / 현재 상태 / 목표 / 비목표 / 완료 기준을 고정 포맷으로 둔다
3. 관련 문서를 절대경로로 연결한다
4. 오래된 문서는 아카이브 또는 경고 표기를 한다

## 최종 원칙

에이전트는 모든 문서를 읽는 것이 아니라,
올바른 문서를 빠르게 찾을 수 있어야 한다.

따라서 좋은 문서 구조는:

- 적은 문서 수가 아니라
- 좋은 역할 분리와
- 좋은 인덱스와
- 좋은 읽기 순서를 가진 구조다
